## 1. 今回の合意事項

### 1.1 ローカル実行・配布
- 成果物は macOS アプリ（`.app`）とする
- `xcodebuild` 等で生成可能であること
- 上級エンジニアの Apple ID で署名すること
- 社内ネットワークの掲示板経由で配布すること
- 各自の Mac 上で実行できることを要件とする

### 1.2 状態遷移
- 埋め込み未実行・クラスタリング未実行状態は、保存済みリストの有無で判別する
- API キー未設定時は、関連操作を無効化してユーザーへフィードバックする

### 1.3 長文レコードの扱い
- 埋め込み入力は 8K トークン相当を上限とする
- 日本語は暫定的に「1文字 = 1トークン」で近似する
- 上限を超過しそうな場合は後方を切り捨てる
- 本件はデモ用途であり、厳密な意味保持よりも単純性・再現容易性を優先する

### 1.4 トピック生成
- データセット全体説明によるグラウンディングを行う
- 隣接トピックの例示により、差異が際立つようシステムプロンプトで誘導する
- デモ用途のため、高度な再現性・厳密性は必須要件としない

### 1.5 3D可視化
- 対応OSは macOS Tahoe 26 以上を前提とする
- 3D座標生成方式は PCA を用いる
- 未対応環境への代替表示は今回のスコープ外とする

## 2. CTO補佐からの整理・補足

### 2.1 ローカル実行要件
「Webサーバー不要」「署名済みアプリ」「社内掲示板で配布」「各自のMac単体で実行可能」という整理になったため、目的に対して十分に明確である。
上級エンジニアには、配布手順と実行時の制約があれば仕様レビューで明文化してほしい。

### 2.2 状態遷移
UIの単純性として妥当。
ただし、上級エンジニアには以下を確認してほしい。

- データ未読込時
- 埋め込み対象列未選択時
- API キーはあるが接続確認未実施時
- 保存済み埋め込みはあるが現在表示中データセットと不整合な時

これらを仕様上の状態として明文化できるか。

### 2.3 長文レコード
デモ優先の割り切りとして妥当。
仕様書では、「1文字 = 1トークン」は厳密値ではなく簡便な近似ルールとして採用する旨を明記してほしい。
また、切り捨て方向は後方切り捨てで固定する。

### 2.4 トピック生成
「再現性よりデモ価値を優先する」という判断は明確でよい。
上級エンジニアには以下をレビューしてほしい。

- 代表データ抽出数
- データセット全体説明に使うランダム抽出数
- 隣接トピック例示の与え方
- 生成済みトピックの保存単位と再表示条件

### 2.5 3D可視化
3D可視化を macOS Tahoe 26 以上に固定し、PCA により 3D 座標を生成する方針とする。
この前提で、上級エンジニアには次をレビューしてほしい。

- PCA の入力対象をどの埋め込み集合とするか
- クラスター選択・凡例・代表トピック表示との接続が自然か
- 3D表示が主役ではなく、意味理解の補助になっているか

## 3. 上級エンジニアへの依頼事項

以下の観点で仕様レビューを依頼する。

1. 上記合意事項に曖昧さ・矛盾・不足がないか
2. 状態遷移の仕様が最小限で十分か
3. 長文切り捨て方針がデモ品質を著しく損なわないか
4. トピック生成の指示方針が、クラスター差異の説明として機能するか
5. 3D可視化の扱いが仕様として自然か
6. 仕様で固定すべき事項と設計へ委ねる事項の切り分けが妥当か
