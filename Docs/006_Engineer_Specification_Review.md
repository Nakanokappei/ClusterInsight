## 依頼事項への回答（6項目）

### 1. 合意事項に曖昧さ・矛盾・不足がないか

**概ね明確。以下2点を指摘します。**

#### (a) 署名・配布の実務制約

「上級エンジニアの Apple ID で署名」とありますが、社内配布のみであれば **Developer ID 署名 + 公証（Notarization）** が必要です。公証なしでは Gatekeeper がブロックし、各自の Mac で開けません。これは仕様として「公証済みの署名付きアプリを配布する」と明文化すべきです。

> **仕様への追記提案:** 「Developer ID で署名し、Apple の公証（Notarization）を通した `.app` を配布する」

#### (b) 空レコードの扱い

データに空テキスト（0文字）のレコードが1件存在します。埋め込み処理での挙動を仕様として定義すべきです。

> **仕様への追記提案:** 「テキストが空のレコードは埋め込み対象から除外する」

### 2. 状態遷移の仕様が最小限で十分か

**CTO補佐から確認依頼のあった4状態を含め、以下の状態遷移を提案します。**

| # | 状態 | 条件 | UI動作 |
|---|------|------|--------|
| S0 | 初期状態 | アプリ起動直後 | CSV読み込みのみ有効 |
| S1 | データ読込済み | CSV読み込み完了 | 埋め込み対象列の選択を促す |
| S2 | 列選択済み | 埋め込み対象列を選択済み | APIキー確認 → 埋め込み実行を有効化 |
| S3 | APIキー未設定 | キーが空または未入力 | 埋め込み・トピック生成を無効化、設定を促すメッセージ表示 |
| S4 | 埋め込み完了 | 埋め込み結果が保存済み | クラスタリング実行を有効化 |
| S5 | クラスタリング完了 | クラスター割り当て済み | トピック生成・3D可視化を有効化 |
| S6 | トピック生成完了 | 各クラスターに要約あり | 全機能閲覧可能 |

**補足（CTO補佐の確認事項への回答）:**

- **データ未読込時（S0）:** CSV読み込み以外の操作は無効化
- **埋め込み対象列未選択時（S1）:** 列選択UIを表示、埋め込み実行は無効化
- **APIキーはあるが接続確認未実施時:** API呼び出し時にエラーハンドリングで対応（事前の接続確認ステップは不要と判断。理由：接続確認と実際の埋め込みで二重にAPIを呼ぶ非効率を回避）
- **保存済み埋め込みと現在データセットの不整合時:** レコード数またはハッシュで整合性を検証し、不整合時は再実行を促す

### 3. 長文切り捨て方針がデモ品質を著しく損なわないか

**損なわない。データ分析結果から裏付けます。**

```
500件のテキスト長分布:
  P50:    769文字
  P75:  1,372文字
  P90:  2,294文字
  P95:  3,712文字
  P99:  7,314文字
  8,000文字超: 3件（0.6%）
```

8Kトークン（≒8,000文字）の上限で切り捨てが発生するのは **500件中わずか3件（0.6%）** です。デモ品質への影響は無視できるレベルです。

**ただし1点留意:** 「1文字 = 1トークン」は日本語の実態としてはやや過小評価です。主要なトークナイザ（OpenAI tiktoken 等）では日本語1文字あたり平均1.5〜2トークン程度になることが多く、実効的な上限は4,000〜5,300文字相当になる可能性があります。それでもP95（3,712文字）以下のレコードが大半であるため、デモとしては問題ありません。

> **仕様への追記提案:** 「1文字 = 1トークンは簡便な近似であり、実トークン数はこれより多くなりうる。デモ用途として許容する」（CTO補佐の依頼どおり明記）

### 4. トピック生成の指示方針が、クラスター差異の説明として機能するか

**機能すると判断します。CTO補佐から求められた4項目を以下に回答します。**

#### (a) 代表データ抽出数
- **提案: 各クラスターから5件**
- 理由：LLMに要約を依頼する際、多すぎるとプロンプトが膨大になりコスト増。少なすぎるとクラスターの特徴を捉えきれない。5件は実用的なバランス。
- **固定値か設計委任か:** 仕様では「各クラスターから少数の代表レコード（目安5件）を抽出」とし、具体数は設計で調整可能とすることを推奨。

#### (b) データセット全体説明に使うランダム抽出数
- **提案: 20件**
- 理由：500件全体の傾向をLLMに伝えるグラウンディング用。20件あれば対応パターンの多様性を概ね網羅できる。全件はトークン制約上不可能。
- **固定値か設計委任か:** 設計で調整可能とすることを推奨。

#### (c) 隣接トピック例示の与え方
- **提案:** あるクラスターのトピック生成時に、既に生成済みの隣接クラスターのトピック名をシステムプロンプトに含め、「これらとは異なる観点で要約せよ」と指示する。
- 全クラスターを一括生成するのではなく、**逐次生成**（1クラスターずつ、前のクラスターの結果をコンテキストに追加）が差異を際立たせやすい。
- **固定値か設計委任か:** 具体的なプロンプト文面は設計に委任。方針のみ仕様で固定。

#### (d) 生成済みトピックの保存単位と再表示条件
- **提案:** クラスター単位で保存（クラスターID + トピックテキスト）。再クラスタリング実行時に全トピックを破棄して再生成。
- 理由：クラスタリング結果が変わればトピックも無効になるため、部分的な保持は整合性リスクが高い。

### 5. 3D可視化の扱いが仕様として自然か

**方針は自然ですが、PCAの選択について技術的な意見を具申します。**

#### PCA vs 次元削減手法の選択

PCAは線形手法であり、高次元の埋め込み空間（通常1,536〜3,072次元）から3次元に射影すると、**クラスター構造が潰れて視覚的に分離しにくくなる**ことがあります。

- **UMAP** や **t-SNE** は非線形手法で、クラスターの局所構造を保存するため、「クラスタリング結果の可視化」という目的にはより適しています。
- 特にUMAPは計算速度も実用的で、500件程度であれば数秒で完了します。

> **意見:** 仕様として「PCA」と固定するより、「次元削減手法により3D座標を生成する（手法は設計で選定）」として設計フェイズに委ねることを推奨します。PCAで十分な分離が得られればPCAを採用し、不十分であればUMAPに切り替える判断余地を残したいです。

ただし、CTOが依存ライブラリの最小化等の理由でPCA固定を判断されるのであれば、それに従います。

#### CTO補佐の確認事項への回答

- **PCAの入力対象:** 全レコードの埋め込みベクトル全体。クラスター単位ではなく、データセット全体で一括してPCAを適用し、全体の中でのクラスター分布を見せる。
- **クラスター選択・凡例・代表トピック表示との接続:** 3D散布図上で各点をクラスター色で色分けし、凡例にトピック名を表示する構成が自然。選択したクラスターの点をハイライトする連動UIは設計フェイズで検討。
- **意味理解の補助になっているか:** 「クラスターがどの程度まとまっているか」「外れ値がどこにあるか」を直感的に把握する補助として適切。ただし3Dは主役ではなく、トピック要約が主たる成果物であることに同意。

### 6. 仕様で固定すべき事項と設計へ委ねる事項の切り分けが妥当か

**概ね妥当。以下の整理を提案します。**

#### 仕様で固定（変更にはCTO判断が必要）
- 成果物形態: macOS署名済みアプリ（.app）
- 対応OS: macOS Tahoe 26以上
- 処理フロー: CSV → 埋め込み → クラスタリング → トピック生成 → 表示
- トークン上限: 8K（1文字≒1トークン近似、後方切り捨て）
- デモ優先の割り切り（厳密な再現性は必須要件としない）

#### 設計に委任（エンジニア判断で決定可能）

- クラスタリングアルゴリズムの選定
- 次元削減手法の選定（PCA / UMAP 等）←CTOの判断待ち
- 代表データ抽出数・ランダム抽出数の具体値
- トピック生成プロンプトの文面
- UIレイアウト・操作フローの詳細
- データ整合性検証の具体的実装方式
